# Лабораторна робота №25
#### Тема: Інтеграція патернів (Factory Method, Singleton, Strategy, Observer)
#### Мета: Розробити систему, що демонструє взаємодію між компонентами, які реалізують патерни Factory Method, Singleton, Strategy та Observer, а також переконатися в коректності їхньої спільної роботи.

### 1. Теоретичні відомості
Factory Method.
Дозволяє створювати об’єкти через фабрику без жорсткої залежності від конкретних класів.

Singleton.
Гарантує існування лише одного екземпляра класу та надає глобальну точку доступу до нього.

Strategy.
Дозволяє інкапсулювати алгоритми та динамічно змінювати їх під час виконання.

Observer.
Дозволяє об’єктам автоматично отримувати повідомлення про зміни стану іншого об’єкта.

### 2. Опис реалізації системи
2.1 Реалізація Factory Method

Було створено:

- Інтерфейс ILogger

Конкретні реалізації:
- ConsoleLogger
- FileLogger
- Абстрактну фабрику LoggerFactory

Конкретні фабрики:

- ConsoleLoggerFactory
- FileLoggerFactory
Factory Method відповідає за створення об’єктів логування без прив’язки до конкретної реалізації.

2.2 Реалізація Singleton

Клас LoggerManager:
- Має приватний конструктор.
- Містить статичну властивість Instance.
- Зберігає поточну фабрику логерів.
- Надає метод GetLogger().

Таким чином забезпечується централізоване керування логуванням у всій системі.

2.3 Реалізація Strategy

Було створено:
- Інтерфейс IDataProcessorStrategy

Реалізації:

- EncryptDataStrategy
- CompressDataStrategy
- Клас DataContext, який:

- приймає стратегію через конструктор;
- дозволяє змінювати стратегію методом SetStrategy();
- виконує обробку через метод Execute().

Це дозволяє змінювати алгоритм обробки без зміни коду клієнта.

2.4 Реалізація Observer

Було створено:
- Клас DataPublisher
- містить подію DataProcessed
- має метод PublishDataProcessed()
- Клас ProcessingLoggerObserver
- підписується на подію
- при отриманні повідомлення використовує LoggerManager для логування результату

Таким чином реалізовано механізм автоматичного сповіщення про обробку даних.

### 3. Демонстрація роботи (відповідність критеріям)
Сценарій 1 — Повна інтеграція

Було виконано:
- Ініціалізовано LoggerManager з ConsoleLoggerFactory.
- Створено DataContext з EncryptDataStrategy.
- Створено DataPublisher.
- Створено та підписано ProcessingLoggerObserver.
- Виконано обробку даних.
- Викликано публікацію результату.

Результат:
У консоль виводяться повідомлення про обробку та логування.

Сценарій 2 — Динамічна зміна логера

Після першого запуску:
- фабрику в LoggerManager змінено на FileLoggerFactory;
- повторно виконано обробку та публікацію.

Результат:
- Логування відбувається у файл log.txt замість консолі.
- Система коректно реагує на зміну фабрики під час виконання.

Сценарій 3 — Динамічна зміна стратегії.

Після першого запуску:
- у DataContext змінено стратегію на CompressDataStrategy;
- виконано повторну обробку та публікацію.

Результат:
- Алгоритм обробки змінився без модифікації клієнтського коду.
- Observer продовжує коректно отримувати та логувати дані.


### 4. Висновок

У ході виконання лабораторної роботи було створено інтегровану систему, що поєднує чотири патерни проєктування.

Factory Method забезпечує гнучке створення логерів.
Singleton гарантує централізоване керування логуванням.
Strategy дозволяє змінювати алгоритми обробки під час виконання.
Observer забезпечує автоматичне сповіщення залежних компонентів.

Інтеграція цих патернів дозволила створити гнучку, розширювану та слабо зв’язану архітектуру, що відповідає принципам об’єктно-орієнтованого проєктування.

Проєкт компілюється та працює без помилок.
