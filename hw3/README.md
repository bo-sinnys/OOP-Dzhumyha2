## Принципи ISP та DIP
### 1. Інтерфейсний принцип розділення (ISP, Interface Segregation Principle)

Суть принципу:
Клас не повинен бути змушений реалізовувати інтерфейси, які він не використовує.
Тобто, краще мати багато «вузьких» інтерфейсів, ніж один «широкий» інтерфейс з непотрібними методами.

## 2. Принцип інверсії залежностей (DIP, Dependency Inversion Principle)

Суть принципу:
Високорівневі модулі не повинні залежати від низькорівневих.
Обидва повинні залежати від абстракцій (інтерфейсів).

<img width="675" height="740" alt="image" src="https://github.com/user-attachments/assets/3785e294-8b76-44ab-9b9c-47e598eb7451" />

Переваги DIP:

- Високорівневий код не залежить від конкретних реалізацій.
- Легко підміняти сервіси (EmailService → SmsService) без зміни OrderProcessor.
- Покращує тестування: можна підставляти мок-реалізації.

## 3. Взаємозв’язок ISP та DI

- «Вузькі» інтерфейси (ISP) дозволяють точніше підставляти залежності через DI.
- Кожен сервіс отримує тільки те, що йому потрібно, що спрощує тестування та зменшує ризик побічних ефектів.
- Наприклад, клас, який тільки друкує, може отримувати IPrinter, а не весь IMultiFunctionDevice.

### Висновок:
- ISP та DIP разом підвищують модульність, гнучкість та тестованість коду.
- Вузькі інтерфейси + абстракції → легше рефакторити та підставляти різні реалізації через DI.
