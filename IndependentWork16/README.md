### Самостійна робота №16
### Тема: Схема розподілу відповідальностей модуля (SRP)
Мета роботи

Навчитися застосовувати принцип єдиної відповідальності (SRP) для декомпозиції складного модуля на менші, більш сфокусовані класи, а також створити UML діаграму класів, яка демонструє розподіл відповідальностей між компонентами системи.

### 2. Початкова реалізація (порушення SRP)

Було створено клас PaymentProcessor, який порушує принцип SRP, оскільки виконує декілька різних відповідальностей:
- перевірка платежу
- списання коштів
- логування транзакції
- відправка SMS

### Чому це порушує SRP

Цей клас має кілька причин для зміни:

- зміна правил валідації платежу
- зміна механізму списання коштів
- зміна системи логування
- зміна сервісу SMS

Це робить код:
- складним для підтримки
- важким для тестування
- менш гнучким для розширення

### 5. Висновки

У ході виконання роботи було досліджено застосування принципу єдиної відповідальності (SRP).

Було встановлено, що:
- початковий клас PaymentProcessor порушував SRP, оскільки виконував кілька різних функцій
- після рефакторингу система була розділена на окремі компоненти з чіткими відповідальностями
- використання інтерфейсів дозволило застосувати Dependency Inversion Principle (DIP)

нова архітектура стала більш:

- гнучкою
- зрозумілою
- легкою для тестування
- розширюваною

Таким чином, застосування принципів SRP та декомпозиції значно покращує якість архітектури програмного забезпечення.
